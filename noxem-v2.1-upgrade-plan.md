### OmniRetrieval

- **What it does**: Unified retrieval framework that routes natural-language queries to heterogeneous data sources (SQL, SPARQL, Cypher, document search) by generating native queries per-source, then fuses cross-source evidence via LLM selection.
- **Key innovation**: (1) Long-context LLM source selection -- feeds full source descriptors (schemas, ontologies, corpus summaries) directly to the LLM instead of embedding them, avoiding the "modality gap" where descriptor embedding biases toward sources resembling the query in form. Returns a shortlist of k=3 candidates, not a single commit, so wrong initial choices can be recovered at the evidence-selection stage. (2) Per-source native query generation preserves structural operators (JOIN, traversal, property paths) that are lost when flattening everything into a shared vector space. (3) Cross-source evidence selection as a dedicated pipeline stage -- handles structurally heterogeneous results (rows, triples, passages, paths) through LLM-verbalized consolidation. No trained components; entire pipeline is zero-shot LLM-prompt-driven. 30 stars, MIT license, 2 commits. Paper: arXiv 2605.29250.
- **Noxem overlap**: Noxem already has a two-path search: entity+attribute direct lookup (`getMemoriesByEntityAttr()` in `memory-store.mjs`, line 434) acts like OmniRetrieval's per-source routing -- if the query has a clear entity/attribute, Noxem skips vector search and does a SQL WHERE clause directly. The FTS5+vector hybrid search with RRF fusion (`memory-server.mjs`, lines 925-1070) parallels OmniRetrieval's multi-backend retrieval. The `classifyQueryIntent()` function (line 743) routes queries between FTS-heavy and vector-heavy strategies, analogous to OmniRetrieval's source selection. Bundle search (`bundle-search.mjs`) does cross-layer graph-based evidence fusion like OmniRetrieval's cross-source evidence selection. The `search/feedback` endpoint (line 1132) provides a relevance signal similar to evidence selection's quality assessment.
- **Noxem gap**: (1) No source routing beyond entity-attribute -- Noxem only has two "sources" (FTS5 and vector), and the entity+attribute direct lookup is a simple SQL scan. There is no intelligence that routes a query like "what are the steps for debugging memory leak" to the `procedures` table rather than the `memories` table. OmniRetrieval proves that per-source native queries outperform a one-size-fits-all retriever. (2) No cross-store query generation -- when a query should search procedures, edges, facets, AND memories simultaneously, Noxem runs them sequentially or not at all. Procedures are searched separately (`searchProcedures()` uses LIKE, not FTS5). Entity records and facets are never searched alongside memories. (3) No multi-candidate commitment deferral -- `classifyQueryIntent()` commits immediately to a FTS/vector weight split. If the intent is misclassified, there is no recovery step (OmniRetrieval's evidence selection stage recovers from wrong source selection 67-75% of the time). (4) No structured-context source catalog -- there is no "catalog" of what data sources exist in Noxem with their schemas and capabilities that an LLM could read to make routing decisions. (5) No HyDE-style query rewriting for document search -- OmniRetrieval generates "hypothetical passages" to improve document retrieval; Noxem's multi-query expansion (lines 938-958) generates alternative phrasings but not HyDE passages.
- **Implementation**:
  1. **Source catalog + LLM-driven routing** in `memory-server.mjs`: Register a static catalog of Noxem's internal data sources: `memories` (FTS5+vector, entity+attr), `procedures` (name+steps, LIKE search), `facets` (per-entity attribute abstractions), `entities` (entity records + mention counts), `core_memory` (persona key-value), `memory_edges` (graph traversal). On each `/memory/search` call, when no entity+attribute match is found and the query is complex (3+ tokens, not identifier-like), pass the catalog + query to Brain 2 LLM to select which sources to query. Return a ranked list of source names. This replaces the hardcoded `classifyQueryIntent()` for complex queries with an evidence-aware router. Add `getSourceCatalog()` function (~30 lines) and modify the search branch (lines 962-1035) to call it before dispatching. Cache routing decisions for similar queries (reuse the Tier 2 semantic cache). ~80 lines.
  2. **Cross-store native query dispatch** in `memory-server.mjs`: After source selection, generate per-source queries. For `procedures`: extract action keywords from the query (e.g. "debug" -> search procedures with trigger_context containing "debug"). For `facets`: extract entity name and search `facets` table by `entity_id`. For `entities`: search `entities.canonical_name` for query nouns. For `core_memory`: check if query matches any core block key. Merge results from all queried sources into the RRF fusion alongside FTS and vector results. Add `searchProceduresByContext(query, limit)` to `memory-store.mjs` that uses FTS5 on procedure steps (currently LIKE-only). Add `searchFacetsByEntity(queryEntity, limit)`. Modify the hybrid search branch to iterate over LLM-selected sources. ~150 lines across 2 files.
  3. **HyDE query rewriting** in `memory-server.mjs`: When the search intent is "conceptual" and Brain 2 is available, generate a hypothetical answer passage before embedding the query. Replace the multi-query expansion (lines 938-958) with HyDE: prompt Brain 2 to "write a brief answer to this query as if you had the information" and embed that answer alongside the original query. This gives the vector search a richer semantic target. Add `generateHyDE(query)` async function (~20 lines). Modify the embed+search path to use [queryVec, hydeVec] as dual query vectors in RRF. ~40 lines.
  4. **Evidence re-rank stage** in `memory-server.mjs`: After RRF fusion and before returning results, add a lightweight LLM re-rank step for high-importance queries (where `expand=true` or limit > 20). Prompt Brain 2 with the top-20 candidates and the query; ask it to mark which are truly relevant. Filter out false positives. This is Noxem's equivalent of OmniRetrieval's evidence selection stage. Add `evidenceRerank(query, candidates)` async function. ~60 lines.
  5. **Procedure FTS5 index** in `memory-store.mjs`: Currently `searchProcedures()` uses LIKE queries which miss semantic matches. Add an FTS5 virtual table `procedures_fts` indexing procedure `name`, `description`, `trigger_context`, and step `text`. Add triggers to keep it in sync. This enables procedure search to participate in the cross-store dispatch as a first-class source. ~50 lines + migration v6.
- **Effort**: L -- Source catalog + routing (M, ~80 lines), cross-store dispatch (M, ~150 lines), HyDE rewriting (S, ~40 lines), evidence re-rank (S, ~60 lines), procedure FTS5 (S, ~50 lines). Total ~380 lines across 3 files + 1 migration. The cross-store dispatch is the most architecturally significant piece because it changes the search pipeline from a 2-source (FTS+vector) model to a multi-source model.
- **Impact**: M -- Cross-store dispatch directly addresses the biggest limitation in Noxem's current retrieval: procedures, facets, entities, and core blocks are invisible to the main search pipeline. OmniRetrieval's benchmark shows source selection is the most consequential step, and evidence selection recovers from wrong routing 67-75% of the time. HyDE rewriting improves conceptual search recall. Procedure FTS5 makes procedures a first-class search source. The source catalog + LLM routing is the highest single-impact item but adds Brain 2 latency to every search call; mitigate by caching routing decisions for similar queries.

### RAG-Anything

- **What it does**: All-in-one multimodal RAG framework built on LightRAG that parses heterogeneous documents (PDF, Office, images), extracts text/table/equation content via specialized analyzers, builds a cross-modal knowledge graph linking text to non-text elements, and retrieves via vector-graph fusion with modality-aware ranking.
- **Key innovation**: (1) Cross-modal knowledge graph -- entities extracted from text are automatically linked to images/tables/equations that reference them via "belongs_to" relationship chains and weighted relevance scoring. This preserves the "this screenshot illustrates that paragraph" linkage that pure-text RAG loses. (2) Modality-aware retrieval ranking -- adjusts result weights based on content type and query intent (image-heavy queries rank image-anchored results higher). (3) Concurrent multi-pipeline processing -- text and multimodal content flow through parallel processing paths, reducing latency. (4) VLM-enhanced query mode -- when a vision model is available, retrieved images are encoded as base64 and passed directly to the VLM alongside text context for richer answer generation. 20.9k stars, MIT license, v1.3.1 (May 2026), 412 commits.
- **Noxem overlap**: Noxem's `entities` table + `memory_entities` join table create a knowledge graph that links memories to entities, similar to RAG-Anything's entity extraction linking. The `memory_edges` table with typed relations (supports, supersedes, related_to, etc.) parallels RAG-Anything's cross-modal relationship mapping. Bundle search's graph traversal (Dijkstra on the hit graph) is analogous to RAG-Anything's vector-graph fusion retrieval. The `categorizeText()` function in `embedding-engine.mjs` classifies memory types, which is a simpler version of RAG-Anything's modality classification. The `scene_name` column on memories preserves document hierarchy, similar to RAG-Anything's "belongs_to" chains.
- **Noxem gap**: (1) No multimodal content support -- Noxem is text-only. Memories have no image, table, or equation fields. There is no way to store a screenshot alongside a text memory and link them. If Noxem-ingested content includes an image reference, the image URL is stored but the image is never processed and cannot be retrieved by visual similarity. (2) No cross-modal entity linking -- when a user says "the error in that screenshot" there is no way to link the text memory about the error to the image memory that shows it. RAG-Anything's "belongs_to" chains solve this. (3) No modality-aware search reranking -- `applyRecencyScore()` and `classifyQueryIntent()` treat all memories equally regardless of content type. If Noxem had image/table memories, there is no mechanism to boost image results for visual queries. (4) No specialized content analyzers -- tables, code snippets, and structured data all get the same flat text embedding. RAG-Anything shows that specialized analyzers (table structure extraction, LaTeX rendering, VLM image captioning) improve retrieval quality on non-text content. (5) No VLM integration for query expansion -- when a query references visual content, the current system cannot leverage a vision model to improve retrieval. (6) No document hierarchy preservation -- the `scene_name` column exists but is never populated by the extraction pipeline, and search does not use it for hierarchical context.
- **Implementation**:
  1. **Multimodal memory fields** in `memory-store.mjs` + migration v6: Add columns `content_type TEXT NOT NULL DEFAULT 'text'` (values: 'text', 'image', 'table', 'code', 'equation') and `content_data TEXT DEFAULT ''` (JSON: for image `{url, caption}`, for table `{headers, rows, markdown}`, for code `{language, snippet}`). Add `content_caption TEXT DEFAULT ''` for VLM-generated descriptions. Modify `storeMemory()` to accept `content_type` and `content_data`. Update FTS5 to also index `content_caption` so image/table captions are searchable. ~60 lines + migration.
  2. **Cross-modal edge typing** in `memory-store.mjs`: When storing a memory with a non-text content_type that references an existing text memory (e.g., a screenshot of an error described in a previous memory), auto-create a `memory_edges` row with `relation='illustrates'` (or `belongs_to`, `depicts`). Add `linkMultimodalMemory(textMemId, visualMemId, relation='illustrates')` function. In bundle search, when traversing the hit graph, also follow `illustrates`/`belongs_to` edges to include related non-text memories in results. ~50 lines.
  3. **Modality-aware search reranking** in `memory-server.mjs`: After RRF fusion, check if any result has `content_type != 'text'`. If the query contains visual keywords (e.g., "screenshot", "diagram", "chart", "image", "look", "see", "show"), boost non-text results by a multiplier (1.3x for image, 1.2x for table, 1.1x for code). If the query is text-focused, apply a slight penalty to non-text results (0.9x). Add `modalityBoost(results, query)` function. ~40 lines.
  4. **Structured content embedding** in `embedding-engine.mjs`: When a memory has `content_type='table'`, serialize the table as Markdown before embedding (headers + row data) rather than the raw JSON. When `content_type='code'`, prepend the language identifier and wrap in backtick fences before embedding. When `content_type='image'`, embed the `content_caption` text instead of trying to embed the image URL. These specializations ensure the embedding captures the structural semantics of the content type. Modify `categorizeText()` to detect table/code markers and set `content_type`. ~70 lines.
  5. **Document hierarchy via scene_name** in `memory-pipeline.mjs`: During L1 extraction, when the source memories come from a document (detected via metadata or session_id prefix), set `scene_name` to a hierarchical path like `doc:{filename}:{section}`. During bundle search, use `scene_name` as a grouping key: memories sharing a `scene_name` prefix get a small score bonus (0.05) because they are contextually related. This activates the existing but unused `scene_name` column. ~40 lines.
- **Effort**: M -- Multimodal fields (S, ~60 lines), cross-modal edges (S, ~50 lines), modality reranking (S, ~40 lines), structured embedding (M, ~70 lines), hierarchy via scene_name (S, ~40 lines). Total ~260 lines across 3 files + 1 migration. The migration is the foundation; everything else builds on the new content_type column.
- **Impact**: M -- Multimodal fields unlock a completely new category of memory content for Noxem. Cross-modal edges + modality reranking make those memories retrievable in context. The biggest single win is structured content embedding: tables and code get much better embeddings when serialized in their native format, improving search relevance without changing the retrieval pipeline. Document hierarchy via scene_name is a "free" improvement that activates existing infrastructure. Note: full VLM integration (image captioning, base64 image passing) depends on an external vision model being available and is out of scope for v2.1; the `content_caption` field provides a placeholder for when VLM becomes available.

### CocoIndex

- **What it does**: Incremental data processing framework (Rust core + Python API) that keeps vector search and RAG pipelines fresh by recomputing only changed data (delta processing), tracking byte-level lineage from source to output, and providing sub-second freshness via memoized transforms.
- **Key innovation**: (1) React-like declarative state model -- you declare `Target = F(Source)`, and the engine continuously reconciles, computing only the minimum work needed. (2) Dual invalidation -- both source data changes AND code/transformation changes trigger recomputation of only affected rows. `@coco.fn(memo=True)` skips computation when input+code are unchanged. (3) Byte-level data lineage -- every vector, row, or graph node in the target traces back to its exact source byte (e.g., "handbook.md L42"), enabling auditability and debugging. (4) 80-90% cache hits on re-index, billed as 10x cheaper at scale. 10.2k stars, Apache-2.0, 1,817 commits. Also provides an MCP server (cocoindex-code) for AI coding agents to query indexed codebases via tree-sitter AST chunking.
- **Noxem overlap**: Noxem's embedding backfill queue (`_embedQueue` in `memory-server.mjs`, lines 206-270) is a simple incremental mechanism: memories are stored first, then embeddings are computed asynchronously. This is a primitive form of CocoIndex's "declare target, engine computes." The semantic query cache (`_queryCache`, lines 273-450) with tier 1 exact + tier 2 cosine hit is a memoization pattern similar to CocoIndex's `@coco.fn(memo=True)`. The `memory_raw` table preserves original text alongside processed/summarized memories, analogous to CocoIndex's source-data preservation. The `citation_log` table tracks which memories were cited in which sessions, a lightweight form of lineage.
- **Noxem gap**: (1) No incremental re-embedding when text changes -- when a memory's text is updated (via supersession, compression, or type correction), the embedding is NOT recomputed. The old embedding remains in `memory_vecs` and in the `memories.embedding` BLOB, serving stale vectors. CocoIndex would detect the source change and recompute only the affected embedding. (2) No code-change-triggered recomputation -- if `categorizeText()` or `estimateImportance()` logic is updated in a Noxem upgrade, existing memories retain their old type/importance. There is no mechanism to re-evaluate memories against the new logic. CocoIndex handles this via `hash(input) + hash(code)` invalidation. (3) No data lineage -- there is no way to trace from a search result back to the exact source conversation turn or session that produced it. The `source_memory_ids` column exists but is only populated by the pipeline (L1 atoms reference L0 episode IDs). Regular memories have no lineage to their origin. (4) No delta-aware sync -- the `/memory/sync` endpoint (line 1792) processes the entire input each time, even if most of it is unchanged. CocoIndex would only process the diff. (5) No embedding model versioning -- if the embedding model is changed (e.g., upgrading `embeddinggemma-300m` to a newer model), there is no detection that all existing embeddings are now inconsistent with new queries. CocoIndex's dual invalidation would flag this and trigger a full re-embed. (6) No cache invalidation on source change -- the query cache invalidates on entity+attribute overlap with new stores, but if an existing memory's text is corrected or its embedding is re-computed, the cache is not selectively invalidated for queries that returned that memory.
- **Implementation**:
  1. **Stale embedding detection + auto-re-embed** in `memory-store.mjs`: Add a `text_hash TEXT` column to `memories` (migration v6). In `storeMemory()`, compute `text_hash = sha256(context_prefix + text).substring(0,16)`. Add a maintenance task in `memory-maintenance.mjs` that scans for `memories WHERE has_embedding = 1 AND text_hash != current_hash`. When found, re-embed the memory and update both `memories.embedding` and `memory_vecs`. Detects when text has changed since last embed (via supersession or compression). Flag `has_embedding` on the memories table too. ~100 lines + migration.
  2. **Logic-versioned re-evaluation** in `memory-maintenance.mjs`: Store a `logic_version TEXT` column in a metadata table `noxem_meta(key TEXT PRIMARY KEY, value TEXT)`. On each server startup, compute `logic_version = sha256(categorizeText.toString() + estimateImportance.toString()).substring(0,8)`. If it differs from the stored version, queue a re-evaluation: re-run `categorizeText()` and `estimateImportance()` on all active memories and update if changed. This is CocoIndex's "code change triggers recomputation" pattern. Increment the logic_version in `noxem_meta` after re-evaluation completes. ~80 lines + metadata table.
  3. **Source lineage tracking** in `memory-store.mjs`: Add `source_type TEXT DEFAULT 'conversation'` and `source_ref TEXT DEFAULT ''` columns to `memories`. When a memory is created from a conversation turn, set `source_type='conversation'` and `source_ref` to `{session_id}:{turn_index}`. When created by the pipeline (L1 extraction), `source_type='pipeline_l1'` and `source_ref=source_memory_id`. When created via `/memory/sync`, `source_type='sync'` and `source_ref=sync_batch_id`. This enables "why does this memory exist?" debugging. Expose via `GET /memory/:id/lineage/source`. ~60 lines + migration.
  4. **Delta-aware sync** in `memory-server.mjs`: Modify the `/memory/sync` endpoint to accept an optional `sync_id` parameter. On first sync with a given `sync_id`, store the hash of the full input. On subsequent syncs with the same `sync_id`, compare the new input hash with the stored one; if unchanged, return immediately (no-op). If changed, compute the diff (added/removed/modified items) and only process the delta. Store the new hash. This prevents re-processing entire conversation histories on each sync call. ~80 lines.
  5. **Embedding model version gate** in `embedding-engine.mjs`: On startup, compute `model_version = sha256(MODEL_ID + EMBED_DIM + DTYPE).substring(0,8)`. Store in `noxem_meta`. If it differs from the stored version (e.g., model was upgraded), log a warning and schedule a full `reembed` of all memories. This prevents the silent inconsistency where old embeddings and new query embeddings come from different models. Already partially handled by `EMBED_DIM` mismatch detection in `vector-index.mjs` (lines 18-26), but that only checks dimension, not the model itself. ~40 lines.
  6. **Selective cache invalidation on memory mutation** in `memory-server.mjs`: When a memory's embedding is updated (via re-embed or text change), find all query cache entries whose `resultIds` contain that memory's ID and evict them. Currently `_invalidateCacheByEntityAttr()` invalidates by entity overlap, which is too broad. Add `_invalidateCacheByMemoryId(memoryId)` that scans `_queryCache` entries for the memory ID in their `resultIds` set. ~30 lines.
- **Effort**: M -- Stale embed detection (M, ~100 lines), logic versioning (M, ~80 lines), source lineage (S, ~60 lines), delta sync (M, ~80 lines), model version gate (S, ~40 lines), selective cache invalidation (S, ~30 lines). Total ~390 lines across 3 files + 2 schema changes (migration v6 + noxem_meta table). The stale embedding detection is the highest-priority item because stale embeddings are a correctness bug that degrades search quality silently.
- **Impact**: L -- Stale embedding detection is the single highest-impact finding from CocoIndex for Noxem. Currently, when memories are superseded, compressed, or type-corrected, their embeddings are never updated. This means vector search returns results based on stale text, which is the #1 cause of "I can see the memory but search doesn't find it" complaints. Logic versioning catches the same problem at the code level (upgrades silently invalidate all computed types/importance). Source lineage enables debugging and auditing. Delta sync reduces the cost of repeated `/memory/sync` calls from O(n) to O(delta). Model versioning prevents the worst-case scenario: switching the embedding model and having no way to detect the resulting inconsistency across the entire corpus. Selective cache invalidation improves cache accuracy after mutations.

### Memary

- **What it does**: Open-source memory layer for autonomous agents that models human-like memory via a Knowledge Graph + Entity Knowledge Store + Memory Stream, tracking entity frequency/recency to rank knowledge depth.
- **Key innovation**: Dual-axis memory model: Memory Stream (breadth -- every entity seen, with timestamps) + Entity Knowledge Store (depth -- frequency + recency ranking per entity). Multi-hop reasoning on KG: extracts key entities from query, builds subgraph of depth 2, joins multiple subgraphs for complex questions. Context window assembly: agent response + top-ranked KG entities + summarized chat history with eviction. Python-based (LlamaIndex + Streamlit dashboard). Supports FalkorDB and Neo4j for KG backend; Ollama for local models (Llama 3, LLaVA). ReAct routing agent with search/vision/location/stocks tools. Multi-graph support for different agent identities. 2.6k stars, MIT license, v0.1.5 (Oct 2024, 250 commits). Released on PyPI (`pip install memary`). Streamlit dashboard visualizes user preferences, entity knowledge depth, and memory evolution. "Rewind Memories" feature (coming) to reconstruct agent state at any historical point.
- **Noxem overlap**: Noxem already has entity tracking via the `entities` table (with `mention_count`, `touchEntity()`, `listEntities()`) and `memory_entities` join table. The `facets` + `facet_points` tables serve a similar role to Memary's Entity Knowledge Store. The cone layer pipeline (L0-L3) provides progressive abstraction that parallels Memary's KG depth traversal. `incrementRecallCounts()` + `boostUsedMemories()` track access frequency. Bundle search traverses memory edges like Memary's subgraph retrieval. The supersession/lineage chain partially overlaps with Memary's Rewind concept (Noxem tracks `superseded_by` + `valid_from`/`valid_until` for temporal validity, enabling `GET /memory/:id/lineage`).
- **Noxem gap**: (1) No explicit recency+frequency combined ranking for entities -- `mention_count` tracks frequency but there is no recency weighting or composite score that combines both. (2) No entity-centric context window assembly -- when serving results, Noxem does not explicitly prioritize "what entities does this user know most deeply." (3) No multi-hop reasoning -- bundle search uses Dijkstra shortest path on direct edges, but does not extract key entities from a query and then traverse the KG at depth 2 to gather neighboring context. (4) No memory stream timeline analysis -- no way to see "when did I interact with entity X most" or detect topic shifts over time. (5) No chat history eviction with summary replacement -- Noxem's `/memory/compress` is all-or-nothing, not a sliding eviction rate. (6) No user preference tracking derived from entity interaction patterns. (7) No rewind/snapshot feature -- cannot reconstruct memory state at a past timestamp (Noxem has lineage but not full temporal versioning). (8) No dashboard for visualizing entity knowledge depth or memory evolution over time.
- **Implementation**:
  1. **Entity recency+frequency composite score** in `memory-store.mjs`: Add `last_mentioned_at TEXT` column to `entities` table (migration v6). In `touchEntity()`, update both `mention_count` and `last_mentioned_at`. Add `getEntityRanking(limit)` function that computes `score = frequency * recency_weight` where `recency_weight = exp(-age_days / half_life)` (default half_life=30d). Return ordered list. ~60 lines + migration.
  2. **Entity-centric search boost** in `memory-server.mjs` `applyRecencyScore()`: After Weibull scoring, apply an entity-boost multiplier. Fetch the top-N entities by the composite ranking from step 1. If a search result's entity is in the top-N, multiply its score by (1 + 0.1 * entity_rank_inverted). This rewards memories about entities the user interacts with most. ~30 lines.
  3. **Multi-hop graph expansion** in `bundle-search.mjs`: After step 1 (multi-layer vector search), extract key entities from the query using the existing `extractEntityAttribute()`. For each entity, run `traverseMemoryGraph()` at depth 2. Add discovered neighbors to the hit graph before Dijkstra. This fills in context that vector search alone misses -- memories 2 hops away that share an entity with the query. ~50 lines.
  4. **Topic shift detection** in `memory-maintenance.mjs`: During maintenance, group memories by `entity` and `created_at` (binned by day). Detect entities whose mention frequency changes by >2x between consecutive time bins. Flag as "shift" and emit a maintenance log. Expose via `GET /memory/topics/shifts`. Future: could auto-trigger L2 extraction or create a "topic shift" memory. ~80 lines.
  5. **Eviction-based context assembly** in `advisor-engine.mjs` `analyzeBeforeCompress()`: Instead of compressing all history at once, implement a sliding eviction model: keep the last N turns verbatim, replace older turns with a ~50-word summary. Calculate eviction rate based on token budget vs total history size. This preserves recent detail while summarizing older context -- matching how Memary constructs context windows. ~70 lines.
  6. **Memory stream endpoint** in `memory-server.mjs`: Add `GET /memory/stream` that returns a chronological log of entity mentions with timestamps, mimicking Memary's Memory Stream. Query the existing `memories` table grouped by entity and `created_at`, ordered chronologically. Add optional `entity` filter param. ~40 lines.
- **Effort**: M -- Entity ranking (S), entity boost (S), multi-hop expansion (S), topic shift detection (M), eviction context (M), memory stream endpoint (S). Total ~330 lines across 3 files + 1 migration. The entity ranking is the foundation; everything else builds on it.
- **Impact**: M -- Entity ranking gives Noxem a "what does this user care about" signal that is currently missing. Multi-hop expansion directly improves bundle search recall (finding memories that vector search misses). Topic shift detection enables proactive memory management. Eviction-based context assembly is a token-saving improvement for Brain 2 calls. The biggest single impact is entity-centric search boost: it makes the system feel personalized because frequently-discussed entities rank higher. The memory stream endpoint adds a missing observability dimension.

### MemPalace

- **What it does**: Hierarchical spatial memory system (Wing -> Room -> Drawer) that achieves 96.6% R@5 on LongMemEval (raw, no LLM; 98.4% hybrid v4 held-out; 99%+ with LLM rerank) by scoping vector search through structural tiers before computing similarity, eliminating cross-domain semantic interference.
- **Key innovation**: Cumulative filtering pipeline: Query -> Wing filter (project/person) -> Room filter (topic) -> ChromaDB vector search -> original text in Drawer. Wing+Room filtering yields 94.8% R@10 vs. 60.9% with flat search -- a 34 percentage-point gain from structural scoping alone. Cross-domain interference (auth discussions across different projects with similar vectors) is the dominant error source; Wing filtering eliminates it entirely. **5-level hierarchy**: Wing (project/person) -> Hall (cross-wing type corridors: facts/events/discoveries/preferences/advice) -> Room (topic within wing) -> Closet (compressed summary pointing to Drawer) -> Drawer (verbatim original text). **Tunnels** cross-link the same room across different wings automatically. AAAK compression language (~30x token compression, natively readable by any LLM without a decoder) used for agent diaries. 4-layer wake-up stack: L0 identity (~50 tokens) + L1 AAAK facts (~120 tokens) = ~170 tokens total at session start vs. ~650K for summarization-based systems. Local-first with embeddinggemma-300m (multilingual, ~300MB) or all-MiniLM-L6-v2 (English, ~30MB). 53.4k stars, MIT license, v3.3.5 (May 2026). 29 MCP tools covering palace reads/writes, KG operations, cross-wing navigation, drawer management, agent diaries.
- **Noxem overlap**: Noxem's cone layers (L0-L3) provide hierarchical organization similar to MemPalace's Closet/Drawer levels. ### LEANN

- **What it does**: Ultra-low-storage vector index that stores graph structure instead of full embeddings, recomputing embeddings on-demand at query time to achieve ~97% storage reduction.
- **Key innovation**: Hub-preserving graph pruning (top 2% high-degree nodes keep full connections) + two-level search with PQ-compressed approximate queue + dynamic GPU batching for recomputation. Published at MLSys 2026 (arXiv:2506.08276). Achieves 90% top-3 recall under 2s with <5% of raw data storage.
- **Noxem overlap**: Noxem already has HNSW-style KNN via sqlite-vec (`vector-index.mjs`) and a graph edge system for memories. The hybrid backend (sqlite-vec + TurboVec sidecar) is architecturally similar to LEANN's two-level search. Embedding dimension is already reduced to 256d (MRL) in `embedding-engine.mjs` to cut storage.
- **Noxem gap**: Noxem stores every embedding as a BLOB in SQLite (`memories.embedding` column) AND in the sqlite-vec `memory_vecs` table -- full duplication. There is no embedding eviction or recompute-on-search. No graph pruning on the HNSW index (all vectors stored equally). No PQ (product quantization) for cheap approximate distance computation. No hub-node caching strategy (all memories cost the same to search). The `findDuplicates()` brute-force is O(n^2) for <500 memories and still expensive for larger sets.
- **Implementation**:
  1. **Embedding eviction layer** in `vector-index.mjs`: Add `evictEmbeddings(options)` function that drops `memories.embedding` BLOB for low-importance, low-recall-count memories while keeping the text + sqlite-vec rowid entry. Store a `has_embedding` flag. On search, when a memory lacks an embedding, call `embed()` on its text to recompute. This cuts storage by ~60% for long-running instances. New env: `EMBEDDING_EVICTION_THRESHOLD_DAYS=60`, `EMBEDDING_EVICTION_MIN_RECALL=2`.
  2. **Hub-node caching** in `vector-index.mjs`: After `initVectorIndex()`, run a one-time analysis: find top-K memories by `recall_count` (proxy for hub-ness). Mark these as `is_hub=1` in a new column. Always keep their embeddings resident. This means ~10% of embeddings cached yield ~40% cache hit rate during search (matching LEANN's finding of 41.9% hit rate at 10% cache).
  3. **PQ-approximate pre-filter** as new function `knnSearchPQ()` in `vector-index.mjs`: Before full KNN, compute PQ codes for all vectors at index time (8-bit quantized, 8 subspaces). Use PQ distance for cheap first-pass, then only recompute full cosine for the top `a*topK` candidates. Store PQ codes in a new SQLite table `memory_pq_codes(id INTEGER PRIMARY KEY, pq_code BLOB)`. This reduces search cost from O(N*d) to O(N*8) for the approximate pass.
  4. **AST-aware chunking** for code memories (adapted from LEANN): In `embedding-engine.mjs` `categorizeText()`, when type is "project" or text contains code fences, split on AST boundaries (function/class level) instead of naive substring. Use tree-sitter-wasm or regex-based function splitting. This produces better embeddings for code memories and avoids splitting mid-function.
  5. **Dedup optimization**: Replace `findDuplicates()` brute-force with KNN-based dedup for all sizes (not just >=500). Use `knnSearchHybrid()` to find nearest neighbor per memory, then only compute full cosine for candidates above a looser threshold (0.85). Reduces dedup from O(n^2) to O(n * log(n) * k).
- **Effort**: M -- Embedding eviction (S, ~100 lines), hub caching (S, ~80 lines), PQ pre-filter (M, ~200 lines + schema migration), AST chunking (M, ~150 lines + tree-sitter dep), dedup optimization (S, ~40 lines). Total ~570 lines across 3 files.
- **Impact**: M -- Storage reduction is the biggest win for long-running instances (months of memories). PQ pre-filter speeds up search 5-10x for large corpora. AST chunking improves code-search relevance noticeably. Hub caching is a free 1.5x search speedup for frequently-accessed memories.

  5. **Proxy mode** in `memory-server.mjs`: Add an optional `--proxy` CLI flag that starts an OpenAI-compatible proxy on a separate port. Intercepts `/v1/chat/completions`, runs content-type compression on the messages, forwards to the real LLM, and passes through the response. ### ReasoningBank

- **What it does**: Strategy-level memory framework for LLM agents that distills reusable reasoning patterns from both successful and failed task trajectories, enabling self-evolving behavior across sequential tasks.
- **Key innovation**: Three-part design: (1) Structured reasoning memories with title/description/content fields extracted via LLM-as-judge self-assessment of trajectories (both successes AND failures). (2) Memory-aware Test-Time Scaling (MaTTS) -- parallel scaling generates k independent trajectories for contrast, sequential scaling iteratively refines, both feed back into richer memory. (3) Virtuous loop: better memory -> more effective scaling -> richer experiences -> higher-quality memory. Accepted to ICLR 2026. Benchmarks: +8.3% on WebArena, +4.6% on SWE-Bench-Verified (~3 fewer steps). Google Research (Sirui Ouyang et al., 17 authors). 61 GitHub stars on `thakshak/ReasoningBank`, MIT license, Python, ChromaDB or JSON backend. Key finding: "memory-driven experience scaling as a new scaling dimension" -- agents self-evolve with emergent behaviors.
- **Noxem overlap**: Noxem's `procedures` + `procedure_steps` + `procedure_context_points` tables are analogous to ReasoningBank's structured reasoning patterns (title, trigger_context -> title/description, steps -> content). The `memory-extract.mjs` module does post-session extraction from conversations (success-oriented). The `advisor-engine.mjs` Brain 2 module does drift detection and session-end analysis, similar to ReasoningBank's self-judgment loop. The cone pipeline's L0->L1 extraction distills raw episodes into facts/preferences -- structurally similar to ReasoningBank's trajectory-to-memory distillation. `boostUsedMemories()` and `incrementRecallCounts()` track which memories are useful, similar to ReasoningBank's retrieval-boosting cycle.
- **Noxem gap**: (1) **No failure-based reasoning memory** -- Noxem only extracts from successful interactions (facts, preferences, goals). It never systematically extracts "what went wrong and why" from failed task attempts. This is ReasoningBank's key differentiator. (2) **No strategy-level memories** -- Noxem's procedures are operationalized steps ("do X then Y") but there is no memory type that captures reasoning strategies ("when encountering infinite scroll, always verify current page identifier first"). (3) **No trajectory-to-memory distillation** -- Noxem stores conversation turns as L0 episodes and extracts factual atoms as L1, but never distills HOW the agent solved a problem (the reasoning trace) into a reusable strategy. (4) **No multi-trajectory contrast** -- Noxem has no mechanism to generate multiple solution attempts for the same task and contrast success vs failure to extract anti-patterns. (5) **No LLM-as-judge self-assessment** -- No memory extraction examines whether a task outcome was good or bad before storing; all extracted memories are treated equally. (6) **No reasoning-memory retrieval at task time** -- when a new task arrives, Noxem retrieves fact/preference memories but not "how to approach this type of problem" memories. ### LESSON.txt -- RAG & Memory Architecture Lessons (Consolidated)

- **What it does**: A curated collection of production-proven RAG and agent memory lessons, covering effective embedding dimensionality, memory governance, memory poisoning prevention, table-aware chunking, HNSW trade-offs, memory-layer architecture, embedding model selection, chunking strategy, and hybrid search patterns.
- **Key innovation**: Three ideas stand out: (1) **Effective dimension (d_eff)** -- theoretical proof that only ~80-90 of 3072 embedding dimensions carry signal (97% is noise), and that compression strategies must adapt to d_eff rather than raw dimension. (2) **Memory judgment > memory storage** -- the mem0 audit (10,134 entries, 97.8% junk, 52.7% system prompts stored as memory, 5.2% hallucinated profiles) proves that write-pipeline governance (orient -> think -> act -> check -> commit) matters more than retrieval speed. (3) **Table handling in RAG** -- tables must be chunked by row structure, not by text length, and require logical table merging across PDF pages with header carry-forward.
- **Noxem overlap**: Noxem already has hybrid search (FTS5 + vector via `embedding-engine.mjs` `searchByEmbedding()` + `memory-store.mjs` `searchMemories()`). The Weibull decay model (`MEMORY_DECAY_HALF_LIFE`) provides temporal validity. `valid_from`/`valid_until` columns on memories and edges provide explicit temporal scoping. The dedup pipeline (`memory-maintenance.mjs` `findDuplicates()`, KNN-based for >=500 memories) addresses data cleanliness. The `importance` scoring + `estimateImportance()` provide basic write-time judgment. Cone layers (L0-L3) provide the multi-layer stack described in the lesson.
- **Noxem gap**: (1) **No effective-dimension awareness** -- Noxem uses 256d MRL embeddings but never measures or adapts to d_eff. The vector index treats all 256 dimensions as equally informative. (2) **No write-pipeline governance** -- `storeMemory()` accepts any text with only auto-categorization and importance estimation. There is no validation gate, no loop detection on repeated stores, no check for system-prompt leakage or hallucinated profiles. The mem0 audit finding (52.7% system prompts stored as memory) directly applies: Noxem has no filter against storing tool output, system messages, or LLM boilerplate as memories. (3) **No memory poisoning protection** -- there is no audit chain, no source tracking with confidence scores, no right-to-forget, and no write validation that rejects obviously poisoned content (e.g., prompt injections masquerading as facts). (4) **No table-aware chunking** -- `memory-pipeline.mjs` extraction treats all text uniformly. If a user stores table data or structured reports, Noxem will chunk by character limit rather than row structure, losing column-header semantics. (5) **No HNSW ef_search tuning** -- sqlite-vec's KNN has no configurable search-accuracy trade-off; `knnSearch()` in `vector-index.mjs` uses whatever sqlite-vec defaults to. (6) **No reranking after retrieval** -- results from FTS5 and vector search are merged with simple dedup (no re-scoring). ### Vercel zerolang

- **What it does**: An experimental systems programming language (C-based compiler, Apache-2.0, 4.8k stars, v0.2.1) designed specifically for AI agents, where the compiler serves as a structured interface between AI and source code. The compiler emits JSON diagnostics with repair metadata, exposes a ProgramGraph for semantic navigation, and supports checked graph edits that validate before mutating source. 816 commits, 312 forks.
- **Key innovation**: **Compiler-as-AI-interface** -- instead of agents parsing text logs, the compiler returns structured JSON with: stable diagnostic codes (`NAM003`), expected vs actual fields, repair identifiers (`declare-missing-symbol`), and typed fix plans via `zero fix --plan --json`. **Checked graph edits** -- agents submit semantic operations (not text patches) that are validated against a graph hash before source is rewritten, preventing stale-context edits. **Skill system** -- compiler bundles version-matched rule text (`zero skills get language`) so agents always read rules matching their binary version. **ProgramGraph** -- exposes modules, imports, symbols, capabilities, effects, and ownership as a navigable semantic graph instead of requiring text-based import inference.
- **Noxem overlap**: Noxem's `procedures` + `procedure_steps` tables are vaguely analogous to zerolang's skill system (versioned operational knowledge). The `memory_edges` typed graph is structurally similar to zerolang's ProgramGraph edges (import edges, module edges). The cone layer pipeline already extracts structured knowledge from raw text, similar to how zerolang's compiler extracts semantic structure from source. The RLM sidecar's multi-call decomposition is architecturally similar to zerolang's compiler-mediated edit loop.
- **Noxem gap**: (1) **No structured diagnostics for memory operations** -- when Noxem operations fail (store, search, pipeline extraction), errors are returned as plain text strings. There are no stable error codes, no repair metadata, no expected/actual fields, and no machine-readable fix suggestions. An LLM calling Noxem MCP tools cannot programmatically understand or fix errors. (2) **No versioned rule/skill system** -- the L1 extraction prompt, the categorization rules, and the importance estimation heuristics are all hardcoded constants in `embedding-engine.mjs`. If these change, there is no way for an agent to know what rules the current Noxem version uses. (3) **No content-hash validation on writes** -- `storeMemory()` does not check if the agent's view of existing memories is stale before writing. There is no "graph hash" equivalent that prevents an agent from creating contradictory memories based on outdated context. (4) **No semantic graph of the memory corpus** -- while `memory_edges` exists, there is no ProgramGraph-equivalent that exposes the memory corpus as a navigable semantic structure (module boundaries, capability declarations, effect annotations). Agents can only search or traverse linearly. (5) **No repair plan generation** -- when maintenance detects duplicates/contradictions, it silently supersedes or archives. There is no structured repair plan that the agent can review before execution. (6) **No dependency graph for procedures** -- procedures are flat with no import/capability/effect metadata, making it impossible for an agent to understand dependencies between learned procedures.
- **Implementation**:
  1. **Structured error diagnostics with repair metadata** across all MCP tools in `mcp-server.mjs`: Define a `NoxemError` class with fields: `code` (stable string like `STORE_001`), `message` (human-readable), `expected` (what was expected), `actual` (what was received), `repair` (object with `id` and `description`, e.g., `{id: 'reduce-text-length', description: 'Split text into smaller chunks'}`). Apply to all tool error paths. Example: `STORE_001` = text exceeds 10MB limit, repair = split into chunks. `SEARCH_001` = no results for query, repair = try broader intent. `PIPELINE_001` = L1 extraction failed, repair = retry with fewer L0 memories. Also add a `memory_explain_diagnostic` MCP tool that accepts a diagnostic code and returns detailed explanation + fix instructions (like `zero explain`). ~180 lines.
  2. **Versioned skill system** in new file `server/noxem-skills.mjs`: Create a skill registry that ships with Noxem version. Skills define: categorization rules, extraction prompt templates, importance thresholds, decay parameters, dedup logic. Expose via `GET /memory/skills` endpoint and MCP tool `memory_skills_list`. When Noxem is upgraded, skills are updated atomically. Agents call `memory_skills_list` at session start to learn current rules (like `zero skills get language`). This prevents agents from operating on stale assumptions about how Noxem categorizes or scores memories. ~120 lines.
  3. **Content-hash validation on store** in `memory-store.mjs` `storeMemory()`: Add optional `expected_hash` parameter. If provided, compute SHA-256 of the entity+attribute combination for the target entity and compare. If the hash differs (entity state changed since the agent last read it), return `CONFLICT_001` diagnostic with the current hash and a suggestion to re-read before storing. This prevents stale-context writes, directly inspired by zerolang's `graphHash` precondition on edits. ~60 lines.
  4. **Structured repair plans for maintenance** in `memory-maintenance.mjs`: Instead of immediately executing dedup/contradiction/decay operations, generate a repair plan first: `[{operation: 'supersede', memory_a: id, memory_b: id, reason: 'cosine=0.94, same entity', confidence: 0.94}]`. Expose via `GET /memory/maintenance/plan` endpoint. Execute only when `POST /memory/maintenance/execute` is called (or on a confirm flag). This gives agents visibility and control over what maintenance will do, like zerolang's `zero fix --plan --json`. ~100 lines.
  5. **Procedure dependency graph** in `memory-store.mjs`: Add `procedure_deps(id, procedure_id INTEGER, depends_on_procedure_id INTEGER, dep_type TEXT)` table. When a procedure is created via `/memory/learn`, extract dependency relationships from the procedure steps (if step A references a concept defined in procedure B, add a dep). Expose via `GET /memory/procedures/:id/dependencies` and MCP tool `procedure_dependencies`. This enables agents to understand the dependency graph of learned workflows, like zerolang's `zero graph --json` exposing import edges. ~120 lines + migration.
- **Effort**: M -- Structured diagnostics (M, ~180 lines), versioned skills (M, ~120 lines), content-hash validation (S, ~60 lines), repair plans (M, ~100 lines), procedure deps (M, ~120 lines). Total ~580 lines across 4 files + 1 migration. The structured diagnostics are the foundation (like zerolang's diagnostic system is the foundation for everything else).
- **Impact**: M -- Structured diagnostics is the highest-value item because it transforms Noxem from a tool that returns text errors into a system that returns machine-readable problems with suggested fixes. This directly improves the reliability of LLM-automated memory management: instead of "Internal error processing request" (current MCP error format), the LLM gets `STORE_001: text too long. Repair: split into chunks`. Versioned skills prevent version-mismatch bugs when Noxem is upgraded. Content-hash validation prevents stale-context writes (a real failure mode in long sessions). Repair plans give agents control over maintenance, converting it from a black box into a reviewable process. Procedure dependencies enable safe modification of learned workflows.

### Key Architectural Decisions

1. **Build our own, don't merge** — All improvements are original Noxem implementations inspired by design patterns from the 15 reference projects. No external code merged in.
2. **Additive schema changes only** — Migration v6 adds columns with defaults, never removes or renames existing columns. All existing queries continue to work unchanged.
3. **Brain 2 dependency is optional** — Tool-description injection, HyDE rewriting, and NL2SQL require Brain 2 (QwenProxy). When Brain 2 is unavailable, these features gracefully degrade: static tool descriptions, no HyDE, no NL2SQL.
4. **MCP tool surface grows minimally** — Phase 1 adds 2 new MCP tools (`memory_wake_up`, ambient context in tools/list). Phase 2 adds 4 (`memory_reasoning_recall`, `memory_compaction`, `memory_retrieve_original`, `memory_capsule`). Phase 3 adds 3 (`memory_audit`, `memory_feedback`, `memory_notebook`). Total: 8 existing + 9 new = 17 tools.
5. **Content-type compression is pure JS** — No new dependencies for JSON/code/log compressors. Only AST chunking (Phase 4) might add tree-sitter-wasm.
6. **Structural pre-filtering uses hard filtering** — Wing/Room constraints are WHERE clauses, not soft weights. MemPalace proves that incorrect candidates surviving vector search are worse than missing a candidate. Hard filter eliminates the failure mode.

### Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Migration v6 breaks existing deployments | All columns have DEFAULT values; no column drops. Test by running existing 65-test suite against migrated DB. |
| Write-pipeline governance rejects legitimate memories | 3-gate filter uses conservative thresholds. Add `MEMORY_WRITE_GOVERNANCE=off` env var to disable. Log accept/reject counts to `/memory/write-stats`. |
| Brain 2 latency on search (source routing, HyDE, reranking) | Cache routing decisions. HyDE only for "conceptual" intent. Reranking only for `expand=true` or limit>20. All 3 skip when Brain 2 is unavailable. |
| Structural pre-filtering over-filters (misses relevant memories in other wings) | Cross-wing tunnel detection (Phase 4) creates bridges. Short-term: allow `wing=''` (all wings) as default. |
| Agent-assisted compaction is too complex for simple use | Automatic maintenance still runs as default. Compaction workflow is an opt-in alternative exposed via MCP tool, not a replacement. |
| PQ implementation in pure JS is error-prone | Defer to Phase 4. Current sqlite-vec KNN is adequate for <50k memories. PQ only needed at scale. |
| Tool-description injection overflows token budget | Cap at 4500 tokens (3K full + 1K summary + 500 procedures). Cache for 5 min. Compress further if model context is limited. |
